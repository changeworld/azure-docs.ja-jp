# Azure セキュリティ センターでのセキュリティの警告の管理と対応
このチュートリアルでは、Azure セキュリティ センターの機能を使用してセキュリティの警告の管理と対応の方法について説明します。

> [AZURE.NOTE]このドキュメントの情報は、Azure セキュリティ センターのプレビュー リリースに適用されます。

## Azure セキュリティ センターのセキュリティの警告の管理方法
Azure セキュリティ センターは、真の脅威を検出し、偽陽性を減らすため、Azure のリソース、ネットワーク、およびマルウェア対策とファイアウォールのようなパートナー ソリューションから、自動的にログ データを収集、分析、統合します。警告の優先順位の一覧が、セキュリティの警告として表示されます。

現在の警告は、[セキュリティの警告] タイルで確認できます。各警告の詳細については以下の手順に従います。

1. **Azure セキュリティ センター** ダッシュ ボードに **[セキュリティの警告]** タイルが表示されます。 

    ![データ収集の有効化](./media/security-center-managing-and-responding-alerts/security-center-managing-and-responding-alerts-fig1.png)

2.  図の中のセキュリティの警告発生回数をクリックすると、**[セキュリティの警告]** ブレードが開き、次の図のようにこの警告に関するより詳細な情報が表示されます。
    
    ![データ収集の有効化](./media/security-center-managing-and-responding-alerts/security-center-managing-and-responding-alerts-fig2.png)

このブレードの下部に、各警告の詳細があります。目的の列をクリックして、列ごとに並べ替えることができます。各列の定義を以下に示します。

- **警告**: 警告に関する短い説明。
- **カウント**: 特定の日に検出された特定のタイプのすべての警告の数。
- **検出元**: 警告をトリガーするサービス。
- **日付**: イベントが発生した日。
- **状態**: その警告の現在の状態。状態には次の 3 種類があります。
    - **アクティブ**: セキュリティの警告が検出されました。
    - **破棄済み**: ユーザーによってセキュリティ警告が破棄されました。この状態は、通常、調査したが実際の攻撃が見つからない、または調査済みで危険が軽減された警告に使用されます。

- **重要度**: 重大度を高、中、低で示します。
  

### セキュリティの警告への対応
組織では攻撃の可能性はさまざまな行動で示されます。たとえば、正規のネットワーク キャプチャを実行しているネットワーク管理者は、何らかの攻撃を開始する人と同じように見える可能性があります。他にも、誤って構成されたシステムに起因し、侵入検出システムで多くの偽陽性が発生し、これにより本物のインシデントを見つけることがより困難になることもあります。Azure セキュリティ センターを使用してセキュリティの警告を確認すると、警告の重要度に基づいて処置を講じることができます。

そのためには、対応する警告を選択します。下の図のように、より多くの詳細が含まれた新しいブレードが右側に開きます。

![データ収集の有効化](./media/security-center-managing-and-responding-alerts/security-center-managing-and-responding-alerts-fig3.png)

この場合、トリガーされた警告が、リモート デスクトップ プロトコル (RDP) の疑わしいアクティビティを参照します。最初の列には、攻撃を受けたリソースが表示され、その後にこの攻撃が検出された時間、警告と重要度の状態が示されます。この情報を確認したら、攻撃を受けたリソースをクリックします。新しいブレードが開き、次の例に示すように、次に行うべきアクションの候補がさらに表示されます。

![データ収集の有効化](./media/security-center-managing-and-responding-alerts/security-center-managing-and-responding-alerts-fig4.png)
  
> [AZURE.NOTE]Azure セキュリティ センターが推奨する修復は、セキュリティの警告によって異なり、場合によっては、推奨される修復を実装するためにその他の Azure の機能を使用する必要があります。たとえば、このような攻撃の修復には、[ネットワーク ACL](virtual-networks-acl.md) または [ネットワーク セキュリティ グループ](virtual-networks-nsg.md) ルールを使用してこの攻撃を生成している IP アドレスをブラックリストに指定します。

### 追加のタスク
日付、状態と重要度に基づいて警告をフィルター処理することができます。[セキュリティの警告] ブレードで、[フィルター] をクリックし、次のように必要なオプションを有効にします。

![データ収集の有効化](./media/security-center-managing-and-responding-alerts/security-center-managing-and-responding-alerts-fig5.png)

警告のフィルタリングは、ダッシュボードに表示する範囲を限定する必要がある場合に便利です。たとえば、システム内の潜在的な違反を調査するために、過去 24 時間以内に発生したセキュリティの警告を確認することができます。セキュリティの警告の大部分には、適用すべき推奨事項がありますが、状況によっては、お使いの環境では偽陽性であったり、特定のリソースの通常の動作であるため、警告を無視する必要があります。いずれの場合も、特定の理由から **[無視]** オプションを使用して推奨事項を非表示にすることができます。

タスクを無視するには、追加情報の表示を適用するか、推奨される構成を適用するリソースをクリックします。無視するタスクを右クリックすると、下の図のように **[無視]** オプションが表示されます。

![データ収集の有効化](./media/security-center-managing-and-responding-alerts/security-center-managing-and-responding-alerts-fig6.png)

### アラート

警告のコレクションでは、それぞれの警告に非常に具体的な説明と修復方法があります。Azure セキュリティ センターの警告は、攻撃シナリオに基づいています。次の攻撃シナリオは、Microsoft エンジンによってトリガーされます。

- **ネットワーク データに対するブルート フォース攻撃の検出**: これらは、ネットワーク トラフィック データから学習する Machine Learning のモデルに基づいて検出されます。 
- **エンドポイント データに対するブルート フォース攻撃の検出**: これらは、マシン ログの Azure セキュリティ センターのクエリに基づいて検出されます。これにより、失敗した試行と成功した試行を区別できるようになります。 
- **悪意のある IP アドレスと通信している VM**: これらは、Azure セキュリティ センターが、ボットによりセキュリティ侵害され、コマンド アンド コントロール (C & C) サーバーと通信しているマシン (またはその反対) の発見に基づいて検出されます。
